Two minute tutorial

You create a Spec, which is a plain Java file which by convention has the name of the class whose behaviour we are specifying followed by the word "Spec", so Foo has FooSpec.

Inside the Spec are a bunch of criteria that specify the behaviour we expect. These are just public void methods that start with "should" and which describe the specific behaviour (e.g. CustomerServiceSpec might have shouldFindCustomerBySurname()).

These will typically set up a known environment, execute some behaviour and then verify that stuff happened. There are some static methods in jbehave.framework.Verify that help with this, for instance Verify.that(somethingHappened) and Verify.equal(expectedValue, actualValue).

Our template criteria looks like this:

public void should[DoSomething]() {
	// setup
	
	// execute
	
	// verify
}

The doSomething is highlighted so we know we have to change it to actually describe some criteria.

Specs can be contained in, wait for it, a spec container. This is a plain Java class that implements the SpecContainer interface. This has a single method: getSpecs(), which returns an array of Classes representing the specs to run. Naturally spec containers can contain other spec containers.

Specs and criteria are verified by reflecting into the spec classes and invoking the criteria methods. A Listener is notified whenever something interesting happens.

A single criteria (yes I know, it should be criterion, I'm over it) is verified by a CriteriaVerifier, a spec is verified by a SpecVerifier

So how do we tie this all together? There is a small (2 line) main method inside SpecVerifier that takes a spec class name as its argument and invokes a SpecVerifier on it passing in a TextListener. Run that with jbehave.jar on your classpath. And, um, that's it. So:

java -cp jbehave.jar;your_classpath jbehave.framework.SpecVerifier your.ApplicationSpec
